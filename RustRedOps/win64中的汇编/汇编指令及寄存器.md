# 扩展-cpu的工作原理

cpu与寄存器和汇编指令紧密相关,是理解汇编和寄存器工作原理的基础

win64下,寄存器是8字节是指,寄存器的寻址能力和计算跨度.与cpu一次读取数据大小(16字节)是两个概念,即Register Width寄存器位宽与总线/缓存带宽(bus/cache bandwidth)的物理差异

1. 物理差异:指针宽度/搬运能力.以rip为例
* RIP(8字节):实质是一个指针,8字节宽度决定的是他能指向的地址空间大小(2的64次方大小)
* 指令预取器(Instruction Fetcher):是cpu内部连接内存/缓存的数据总线(data bus).通常是16字节128位的,有些是32字节256位宽的
* 物理动作:cpu根据rip(8字节地址)去指令缓存(L1 cache)抓取数据,其一次抓取数据的多少与bus data有关,即cpu一次性会抓取16/32字节长度的块放入指令预取缓冲(Instruction Prefetch Buffer)

2. 为什么是16字节:是由x64指令集的变长特性决定的
* x64指令最长可以达到15字节(0-16).什么是cpu指令?和汇编指令有啥区别?cpu指令即机器码实质是字节流,但可以被cpu识别
* 如果cpu每次只抓取8字节(Register width),那么当遇到一个15字节的指令时,就无法读全.

3. 指令时如何从16字节中切出来的
想象指令流是一个连续的字节池，RIP 指向池中的某个位置：
* 抓取： CPU 从 RIP 地址开始，往后抓 16 字节
* 扫描： 译码器查看这 16 字节的前几个字节
  *  例如：它看到第一个字节是 0xE8（CALL 指令的操作码）.一个字节不是只有8位吗?这里的0xE8没有超出范围?这个8字节是无符号的范围0-255
  *  计算： 它立刻知道这是一条“1字节操作码 + 4字节偏移”的指令，总长度 `$L = 5$`
*  切分： 译码器从 16 字节中切出前 5 字节送去执行
*  推进： RIP 立即增加 5（RIP = RIP + 5）。下次抓取就会从新的 RIP 开始
 

1. 超宽抓取在红队的应用
“抓取宽度 > 寄存器宽度”的物理特性，是很多高级对抗技术的根源

A. 指令重叠 (Instruction Overlapping)
* 技术点： 既然 CPU 是一次性抓一堆字节，我们可以构造出一些“畸形”的字节序列。
* 对抗： 我们可以让一个 CALL 指令的“身体”里，隐藏着另一个合法的 INC EAX 指令。
* 效果： 静态分析工具可能只看到一种路径，但由于 RIP 的不同对齐，CPU实际执行的是另一种路径。这在 代码混淆（Obfuscation） 中极为常见。


B. 规避 EDR 的硬件断点 (Hardware Breakpoints)
* 某些安全软件会通过 调试寄存器 (DR0-DR7) 监控特定指令。如果你理解了 CPU预取指令的块状特征，你可以通过构造跨越缓存行（Cache Line）的指令流，在某些极端情况下干扰硬件监控的触发时机。


C. 缓冲区溢出的精确计算
* 在编写 puerto 的 Shellcode 时，你必须确保你的 RET 指令正好落在 CPU 抓取的那个 16字节块的起始逻辑位置。如果你的指令被截断在两个抓取块之间，可能会引发微架构层面的性能损耗，甚至在某些开启了内存保护的系统上触发异常

## jmp

无条件跳转指令,强制修改rip的值

1. jmp rbx:跳转到rbx寄存器所指向的内存地址里的那个值
2. jmp 0x12315:跳转到指定内存位置
3. 不同的跳转会有固定的操作码
4. EDR和CPU会监控间接跳转:如果一个jmp跳到一个不属于代码段的区域,或者没有.pdata记录的区域,会博报告异常

`jmp qword ptr [rbx]`:  
1. qword ptr:一个64位的地址数值
2. rbx:要跳转的目的地址存在rbx指向的内存中




## CALL

win64下CALL是执行流转移与栈帧生命周期的发起者(Initiator).不仅实现RIP的跳转,还通过硬件逻辑与ABI的强制耦合,确立后续所有栈操作的物理基准

* 硬件层面的原子微操作 (Hardware Atomic Operations):在 CPU 执行 CALL指令时，硬件逻辑电路会无条件地按序执行以下三个原子动作，且该过程不具备对栈对齐状态的校验能力：
   1. 计算**返回地址** (Calculate Next RIP)： 计算 CALL 指令下一条指令的 64 位虚拟地址(RIP+call指令大小),此时rip以自增指向了下一条指令地址.假设是0x1005
   2. **压栈返回**地址 (Push RIP)： 执行类似 RSP = RSP - 8; `[RSP]` = Next_RIP 的物理写入。这里将0x1005压入栈顶
   3. 加载目标地址 (Load RIP)：将目标函数的起始地址（或寄存器/内存中的间接地址）加载到 RIP 寄存器中。将目标函数的地址假如是0x2000放入rip
   4. 以上三步结果是rip变为0x2000.下一个周期cpu会从0x2000执行指令,此时跳转成功.至于怎么回来的要使用ret(从0x2000回到0x1005)

* 物理结果： 栈顶（RSP）现在物理指向一个 8 字节的返回地址槽位，这成为了被调用函数（Callee）栈帧的逻辑最高边界
* 16n+8:执行call指令瞬间,rsp必须是16字节对齐的,call之后压入了8字节返回地址,进入函数瞬间rsp变为16n+8.此后回到栈上时仍需将rsp转为16字节对齐(因此在所有non_leaf函数序言中,执行sub rsp ,x时,x必须时8的奇数倍)
* 关于此时影子空间:call默认caller在call压入返回地址之前,在栈上预留了4个8字节空间(返回地址物理上紧挨着影子空间的底部,当伪造返回地址时,必须保证返回地址上方有32字节空间)
* CET硬件的缺陷与uwd的生存空间:cet是一个硬件补丁,在cpu内部维护一个不可见/只读的栈.call发生时,地址被同时压入数据栈和影子栈;ret发生了cpu比较两者的值.
* 因此,cet只审计ret.cet的防御逻辑是强耦合在这两对指令上的.那么对于JMP指令不处罚影子栈备份,也不触发影子栈比对,就成为了cet的漏洞







## ret

win64下,RET(return from procedure),是函数生命周期的终结,Epilog的核心,是结构化异常处理(SEH)正常链式工作的物理保障.确保RSP在RET执行瞬间指向正确的返回地址是Epilog唯一且绝对的责任

1. 硬件层面的原子操作 (Hardware Atomic Operations):CPU 执行 RET 指令那一瞬间，硬件自动按序执行两个原子动作：
   * 弹出返回地址： 硬件从当前的栈顶 (`[RSP]`) 读取 8 字节数据，同时执行rsp=rsp+8,将这8字节栈空间释放.此时读取的8字节就是当初call压入的地址.例子是0x1005
   * 加载指令指针(load rip)： 硬件将刚才读取到的8字节直接覆盖写入到rip中.此时rip从原本正在执行的函数地址0x2000处,变回call之后的0x1005

* ret指令在执行时,不检查弹出的地址是不是由之前的call压入的.ret会将rsp指向的位置弹出8字节给rip
* CPU 硬件无法识别 `[RSP]`中存储的是合法的指令地址还是普通的业务数据。若软件未能维持栈平衡，CPU 将会把数据误认为地址加载至RIP，在下一取指周期因执行非法内存而触发 #GP (General Protection Fault) 或 #PF (Page Fault)

## rip

CPU :取指-译码-执行-写回 (Fetch-Decode-Execute-Write-back)

在Fetch阶段,cpu从当前rip指向的地址读取指令字节.当这些字节进入cpu的指令预取缓冲,硬件会根据解码出的指令长度,将rip增加对应长度.  
即,当CALL指令进入execute阶段,rip已经自动指向下一条指令的起始地址  
这意味着:硬件在执行call的压栈操作时,不需要再做加法,它直接把当前rip寄存器里面的值(此时已经是下一条指令的地址了)压入栈即可.

## rsp和rip之间的联系

rip rsp最直接的联系发生在函数调用和返回的瞬间,他们通过栈内存进行数据交换.

核心纽带 call ret硬件指令

**Call**
1. call:rip写入rsp. cpu把rip的当前值即call的下一条指令(也就是返回地址)写入rsp.
2. rsp-8 栈顶下移
3. rip载入新函数入口地址,cpu跳转执行

**ret**
1. rsp写入rip:rsp +8 ,栈顶上移,回到返回地址所在位置
2. cpu从当前rsp指向的内存中读取8字节数据,并强行写入rip
3. cpu顺着新rip地址跳转,回到调用者函数中继续执行






## rbp

* 全称: 64-bit Base Pointer-基址指针
* 物理意义:通常指向当前函数栈帧的起始位置
* 逻辑意义:为函数内部访问局部变量和参数提供了一个固定参考点
  
* RBP在栈中的位置
RBP 位置的核心在于函数序言（Prolog）中的那两行定义
```asm
push rbp ;压栈
mov  rbp,rsp
```
1. push rbp (保护旧现场):RSP向下移动8字节(减法?对不对),把上一个函数的RBP存入栈顶.
* 物理位置:此时栈顶RSP存放的是旧RBP

2. mov rbp,rsp :执行后RBP的值和当前的RSP完全一样
* 物理位置:RBP现在指向刚才压入的那个旧RBP的存储地址

假设call指令执行前,rsp处于一个完美的16字节对齐边界上(假设为 0x...00)  
1. 执行call:自动压入8字节返回地址,rsp的地址不再是16字节对齐的
2. 执行push rbp:手动压入8字节旧RBP.此时RSP重新回到了16字节对齐
3. 执行mov rbp, rsp : RBP的值现在等于 0x...F0(为啥是F0,往下看)  
图示prolog的push rbp  mov rbp, rsp之后的stack布局:

| 物理地址（示例） | 偏移量 | 栈内存内容 (8字节一格) | 物理位置描述与逻辑含义 |
|------------------|--------|------------------------|------------------------|
| 0x...20          | +0x30  | Parameter 5            | 调用者压入的第 5 个参数 |
| 0x...18          | +0x28  | Shadow Space (for R9)  | 预留给寄存器参数的 32 字节空间 |
| 0x...10          | +0x20  | Shadow Space (for R8)  | (影子空间的高端) |
| 0x...08          | +0x18  | Shadow Space (for RDX) | (影子空间的低端) |
| 0x...00          | +0x10  | Shadow Space (for RCX) | 调用者帧的起始点 |
| 0x...F8          | +0x08  | Return Address         | call 指令压入的返回地址 |
| 0x...F0          | +0x00  | Saved Old RBP          | RBP 和 RSP 共同指向这里！ |
| 0x...E8          | -0x08  | (未定义/编译器填充)    | 尚未执行 sub rsp, X，此处是“虚空” |


通过这张图，你可以发现几个极其重要的物理真相：

   1. RBP 的物理指向：
      RBP 此时指向的 不是 局部变量，而是
  “上一个函数留下的坐标原点”。即：[rbp] 里面存的值就是 Old RBP。
   1. 返回地址的绝对位置：
      在 Windows x64 中，返回地址 永远 位于 RBP + 8 的物理位置。这是 EDR
  和调试器进行栈回溯（Unwinding）的硬性物理基准。
   1. 16字节对齐的维持：
      你会发现，RBP 指向的地址（0x...F0）本身就是 16字节对齐
  的。这意味着，如果接下来函数要执行 sub rsp, 20h 分配局部变量，新的 RSP
  依然会保持 16 字节对齐。
   1. 参数 5 的物理距离：
      为什么是 +0x30？
      计算：RBP(0) -> Saved RBP(8) -> ReturnAddr(8) -> ShadowSpace(32) =
  48 字节 ($0x30$)。
      所以，想在函数里通过 RBP 拿到第 5 个参数，物理代码必须写成 mov
  rax, [rbp + 30h]。

  4. 在 uwd 项目中的终极意义

  uwd 项目的核心任务是 “欺骗堆栈验证（Stack Validation）”。


   * 它的伪造逻辑：
      当 uwd 伪造一个帧时，它必须在内存里手动摆放这些数据。
       * 它会在某个地址写下 Old RBP 的值。
       * 它会紧接着在 +8 的位置写下一个合法的 kernel32
         内部地址（伪造返回地址）。
       * 最关键的： 它必须把 RBP 寄存器的值指向它写 Old RBP 的那个地址。


  如果不严谨会怎样？
  如果 uwd 把 RBP 指向了 Saved RBP 的上方或下方 8 字节，那么当 Windows
  尝试执行异常处理或 EDR 扫描时，RBP + 8
  拿到的就不是返回地址，而是影子空间的乱码。系统会立即判定为 “Stack
  Corruption”，直接触发蓝屏或防御报警。


  结论：
  RBP 就是 “栈帧的脊椎”。push rbp; mov rbp, rsp 之后，RBP
  锁定了当前的对齐状态，并为向上访问参数、向下访问变量建立了一个
  绝对物理原点。

* 核心作用:
x86下,RBP强制使用.x64下,rbp是通用寄存器,但其核心作用依然不可替代:  
1. 访问局部变量:不管RSP因为压栈操作变到了哪里,局部变量相对于RBP的偏移永远不变
  * 如`[rbp-0x10]`永远是第一个局部变量.?
2. 访问函数参数:
  * 对于超过4个参数的函数,需要通过栈传递函数的参数
  * 如`[rbp+0x30]`可能是第5个参数.?
3. RBP向上往高地址走(加法)有返回地址\参数等由调用者准备的内容;向下往低地址走(减法),进入当前函数新开辟的空间.这里的局部变量是当前函数序言继续执行sub rsp, x之后产生的
4. 栈回溯 (Stack Walking)：
      这是 uwd 项目最关心的功能。经典的栈回溯是通过 RBP 链实现的：
       * 当前的 RBP 指向旧的 RBP。
       * 旧的 RBP 又指向更旧的 RBP。
       * 这形成了一个清晰的链条，调试器只需顺着 RBP就能还原出整个调用历史。
       * 这些旧的RBP都是在栈上一直保存着的吗?不然为啥能读取旧的RBP内容?



  1. RBP 的关键特性


  A. 非易失性 (Non-volatile)
  在 Windows x64 调用约定中，RBP 是被调用者保护 (Callee-saved) 的。
   * 这意味着：如果一个函数想用 RBP，它必须在序言里先 push
     rbp，在尾声里再 pop rbp。必须保证函数返回时，RBP
     的值和进来时一模一样。


  B. 可选性 (Frame Pointer Omission - FPO)
  现代编译器（如 Rust 的 rustc 或
  MSVC）非常聪明。它们发现：既然我能精确计算出 RSP
  的每一次变动，那我就不需要 RBP 了，直接用 [rsp + offset]
  访问变量就行。
   * 优点： 多出一个通用寄存器供计算使用，提高性能。
   * 后果： 导致“栈回溯”变难了。这也是为什么 Windows 必须引入 .pdata
     表的原因——因为编译器不保证有 RBP
     链，系统必须通过元数据来找回调用者。


  C. 对齐缓冲
  在序言中，mov rbp, rsp 之后，即使 RSP 之后为了对齐又减了 8 字节，RBP
  依然保持着原始进入函数时的位置参考。

  ---

  4. 在 uwd 项目中的灵魂地位


  你正在研究的 uwd 为什么要死磕 RBP？请看 uwd.rs 里的这行代码：


   1 fn find_push_rbp(module_base: *mut c_void, runtime_table:
     &[IMAGE_RUNTIME_FUNCTION]) -> Option<Self> {
   2     // 它在寻找那些手动 PUSH RBP 的函数
   3 }


  为什么 uwd 喜欢带 RBP 的函数？
   1. 稳定性： 带有 push rbp 指令的函数，其 Unwind
      机制通常更标准、更易于模拟。
   2. 欺骗 EDR： 很多高级 EDR 的扫描引擎如果看到栈回溯中缺少 RBP
      链，可能会触发启发式报警。uwd 通过模拟一个带有 RBP
      的合法帧，让栈看起来极其“自然”。
   3. 精确计算： uwd 需要知道
      rbp_offset。因为它要在伪造的栈空间里，在确切的偏移位置填入一个虚假
      的返回地址或上一级 RBP 的值。

  ---

  5. RBP 与 RSP 的爱恨情仇


   * RSP 是“动态”的： 随着 push, pop, sub 不断跳动。它是 CPU
     执行的物理边界。
   * RBP 是“静态”的：
     在一个函数生命周期内通常保持不动。它是程序员理解代码逻辑的逻辑边界
     。


  总结给您的技术视角：


  作为一个红队开发者：
   1. 当你看到 push rbp; mov rbp,
      rsp，你就知道进入了一个“有规矩”的函数，它是你伪造栈的最佳模版。
   2. 当你用 Rust 写 unsafe 汇编时，记住 RBP 是你的锚点。
   3. 如果你想让你的 puerto 隐身，你就得像 uwd 那样，学会如何在没有物理
      RBP 链的情况下，通过修改内核对象手动构建出一套让系统信以为真的逻辑
      RBP 链。


  RBP
  不仅仅是一个存储地址的容器，它是程序执行流的“史书”。掌握了它，你就掌握
  了重写历史的能力。
