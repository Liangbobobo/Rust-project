# Magic Numbers

魔数在winnt.h/windows.h/cpu硬件架构中代表特殊的含义

在windows内核中,绝大多数数字都是位掩码bitmask,利用二进制中特定的1的位置,来表示某种特定的开关

## 内存保护常量

表示内存权限的常量:
1. PAGE_NOACCESS  =  0x01  (二进制  0000 0001 )  
2. PAGE_READONLY  =  0x02  (二进制  0000 0010 )  
3. PAGE_READWRITE  =  0x04  (二进制  0000 0100 )  
4. PAGE_WRITECOPY  =  0x08  (二进制  0000 1000 )  

以上是以2的次方递增,每个权限占用一个独立的二进制位.

表示带执行execute的权限常量(将上面不带执行的权限,整体左移4位,及乘以16):
1. PAGE_EXECUTE  =  0x10  (二进制  0001 0000 )
2. PAGE_EXECUTE_READ  =  0x02 * 0x10 = 0x20  (二进制  0010 0000 )
3. PAGE_EXECUTE_READWRITE  =  0x04 * 0x10 = 0x40  (二进制  0100 0000 )

操作系统在分配内存时,会将这个数值直接送到cpu的mmu(内存管理单元).cpu的页表项(PTE)中有专门的位(如XD/NX位)来控制执行权限/读写位权限(R/W位).  
比如传入0x40,内核九八这个物理内存页映射为底层的硬件全开状态

```rust
pub const PAGE_READWRITE: u32 = 0x04;
pub const PAGE_EXECUTE_READ: u64 = 0x20;
pub const PAGE_EXECUTE_READWRITE: u64 = 0x40;
```

有些是u32有些是u64:
1. 从os底层来看,不管是哪种内存保护权限.在win内核中都是32位(DWORD/u32)
2. hypnus项目中后续把这些内存保护权限的魔数传给了寄存器.但是win64下的寄存器都是64位的.可能是偷懒就直接写为u64,避免后续使用`ctxs[7].R9 = protection as u64;`
3. 这种做法是不规范的,在重构中的定义时应该写成  u32,对应  DWORD.在具体的寄存器赋值逻辑处,统一显示使用as u64进行转换
