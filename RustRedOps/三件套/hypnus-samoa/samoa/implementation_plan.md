# Hypnus 工程架构与依赖关系分析

本文档旨在梳理 `hypnus` 项目中各个源码文件的依赖关系，并为您目前“直接重构 `hypnus.rs` 感到吃力”的问题提供架构级解法与重构先后顺序建议。

## 架构痛点分析：为什么直接重构 `hypnus.rs` 会吃力？

`hypnus.rs` 是整个免杀逻辑的“顶层指挥官”。它内部调用了 `config`（获取底层 API 指针）、`spoof`（构建伪造栈）、`allocator`（堆加密）以及 `cfg`（控制流守卫绕过）。
如果您在底层地基（那些相互纠缠的文件）尚未稳固时直接动顶层建筑，就会遭遇“牵一发而动全身”的代码报错风暴。

通过对源码的分析，我发现底层文件之间存在着**复杂的交叉引用（甚至循环依赖）**，这是导致您重构困难的核心原因：
- `config.rs` 依赖 `spoof::StackSpoof` 和 `cfg::is_cfg_enforced`。
- `spoof.rs` 依赖 `config::Config` 和 `gadget::scan_runtime`。
- `gadget.rs` 依赖 `config::Config` 和 `spoof::Unwind`。
- `cfg.rs` 依赖 `config::Config`。

这种蜘蛛网状的依赖关系，在 Rust 中虽然编译器允许，但对于人工重构是灾难性的。

## 推荐的重构先后顺序（自底向上的解耦策略）

为了顺利重构，我们必须采用**自底向上 (Bottom-Up)** 的策略。先重构没有任何内部依赖的“叶子节点”，再处理复杂的“核心引擎层”，最后重构“顶层编排层”。

### 第一阶段：基石层（无内部依赖）
这些文件只负责定义基础数据结构和向外声明系统 API，不依赖项目内的其他业务逻辑。
1. **`types.rs`**
   - **作用**：定义所有贯穿全局的类型（如 `ObfMode`, `Obfuscation`）和从 C 语言翻译过来的 Windows 结构体。
   - **重构建议**：检查类型安全性，确保 `repr(C)` 和 `repr(transparent)` 使用正确。
2. **`winapis.rs`**
   - **作用**：基于 `dinvk` 或手动声明所需的底层 Syscall 和 NTAPI（如 `NtCreateEvent`, `TpAllocTimer`）。
   - **重构建议**：统一函数签名规范，确认是否需要将硬编码的系统调用剥离。
3. **`allocator.rs`**
   - **作用**：全局自定义堆分配器及堆加密逻辑。
   - **重构建议**：它只依赖基础类型，逻辑非常独立。确保堆的加解密逻辑（`RtlCreateHeap`）线程安全且独立可用。

### 第二阶段：战术引擎层（解决循环依赖）
这是重构的深水区。建议优先剥离逻辑，打破他们对 `config::Config` 全局单例的过度依赖。
4. **`gadget.rs`**
   - **作用**：在内存中扫描合法的 ROP 链节点（如 `jmp rbx`, `ret`）。
   - **重构建议**：目前它依赖 `spoof::Unwind`。建议将 `Unwind` 结构体提取到一个独立的 `pe_utils.rs` 或放入 `types.rs` 中，切断它对 `spoof.rs` 的反向依赖。使其成为纯粹的内存特征码扫描工具。
5. **`spoof.rs`**
   - **作用**：调用栈伪造（Stack Spoofing）的核心逻辑。
   - **重构建议**：在 `gadget.rs` 重构完毕后，`spoof.rs` 就可以安心调用扫描功能。重点重构计算 `UNWIND_INFO` 和栈帧大小的复杂逻辑。
6. **`cfg.rs`**
   - **作用**：CFG（控制流卫士）探测与绕过，用于注册合法的间接调用目标。
   - **重构建议**：逻辑相对简单，在有了底层 API 支持后，将其重构为独立的静态功能函数。

### 第三阶段：全局配置层
7. **`config.rs`**
   - **作用**：全局单例（Singleton），在初始化时一次性解析出所有需要的 API 地址、ROP Gadget 和 CFG 状态。
   - **重构建议**：在前两个阶段稳定后，`config.rs` 只需要像拼图一样把 `spoof`, `gadget`, `cfg` 拼装起来。您可以放心地重构其内部的 `spin::Once` 懒加载逻辑。

### 第四阶段：顶层编排层
8. **`hypnus.rs`**
   - **作用**：暴露给用户的宏（`timer!`, `wait!`, `foliage!`）以及构建 10 步异步 CONTEXT 状态机的所在地。
   - **重构建议**：此时所有底层接口已经极其清晰且稳定。您只需要关注 `Hypnus::timer()` 里面那 10 个 `ctxs` 的 RDX、RCX、RSP 赋值逻辑是否正确，以及 `TpAllocTimer` / `TpSetTimer` 的时序逻辑。
