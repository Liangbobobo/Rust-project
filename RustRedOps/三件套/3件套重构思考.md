# hypnus中能否用debug_log!(puerto)代替bail!

不能,但应当借鉴移除使用anyhow库

1. debug_log!宏是“副作用输出宏”：它的作用是将信息打印到调试器，执行后控制流会继续向下运行
2. anyhow::bail! 宏是“控制流宏”：它的作用是立即构造一个错误并 return Err(...)，提前终止当前函数的执行

绝对不能直接用 debug_log! 替换 bail!，否则会导致本该因为错误而终止的逻辑继续执行，引发灾难性的崩溃

最终选择的方案是debug_log!+enum的模式

## 用栈分配替换hypnus::anyhow::bail!

源码中使用的obfstr,在解密中也可能被扫描到.更加隐蔽方式是puerto中error+debug_log!模式

hypnus 中使用 anyhow 和 obfstr（如 bail!(s!("NtQueryInformationProcess Failed"))）有以下缺陷：
1. anyhow 强制依赖堆分配（alloc），这在高级免杀中是不可接受的.虽然hypnus自定义了堆分配器
2. 即使 s!宏在运行时才解密字符串，但解密后的字符串依然存在于栈或堆中，且带有格式化开销
3. 包含了复杂的虚表分发（Dynamic Dispatch）逻辑，使得最终载荷（Payload）体积臃肿

如前所述,bail!使用anyhow是为了直接中断控制流.为了用debug!+enum复现这个功能.需要
1. 定义全局Error Enum,用于定义与底层系统调用兼容的错误类型.

```rust
#[repr(u32)] // 保证内存布局为 4 字节，与 C/系统底层兼容
pub enum HypnusError {

Success = 0,

 // API/Syscall 获取与调用失败
    6     SyscallFailed = 1,
    7     ApiResolutionFailed,
    8
    9     // 内存与栈相关错误
   10     StackAllocationFailed,
   11     MemoryProtectionFailed,
   12
   13     // ROP/Gadget 相关错误
   14     GadgetNotFound,
   15     InvalidNtHeader,
   16
   17     // 线程与事件相关错误
   18     EventCreationFailed,
   19     ThreadContextFailed,


// 可选：为了让你的函数签名从 anyhow::Result<T> 变为自定义的 Result
pub type Result<T> = core::result::Result<T, HypnusError>;

}

```

第二步：实现定制的 stealth_bail! 宏
  这个宏的作用是：接收一个 Enum 变体和相关的调试字符串。在 Debug
  模式打印日志并返回错误；在 Release 模式直接静默返回 Enum 错误。

    1 #[macro_export]
    2 macro_rules! stealth_bail {
    3     // 匹配模式：(错误枚举, 格式化字符串, 参数...)
    4     ($err:expr, $($arg:tt)*) => {
    5         {
    6             // #[cfg(debug_assertions)] 保证了大括号内的代码在 Release
      模式下完全不参与编译
    7             #[cfg(debug_assertions)]
    8             {
    9                 // 这里调用 dinvk 中基于栈分配输出的 println! 或者标准库的
      println!
   10                 $crate::println!("[!] Error ({:?}): {}", $err,
      format_args!($($arg)*));
   11             }
   12
   13             // 无论 Debug 还是 Release，都会中断当前函数，返回无字符串的
      Enum 错误
   14             return core::result::Result::Err($err);
   15         }
   16     };
   17
   18     // 重载匹配模式：如果只想返回错误，不想写日志
   19     ($err:expr) => {
   20         return core::result::Result::Err($err);
   21     };
   22 }

  完美闭环：使用 From Trait 桥接内部 Enum 与 NTSTATUS

  你需要利用 Rust 的 From 或 Into trait，实现一个从你的自定义 Enum 到标准
  NTSTATUS 的显式转换表。

  这样做的核心优势是：在项目内部你保留了详尽的错误分类（方便开发和逻辑判断），在
  项目外部你完美伪装成了合法的 Windows 底层模块。

  完整实现示例

    1 // 1. 定义 NTSTATUS 类型别名（如果你的项目中还没有的话）
    2 pub type NTSTATUS = i32;
    3
    4 // 2. 定义常用的合法 NTSTATUS 常量（这些是微软官方定义的值）
    5 pub const STATUS_SUCCESS: NTSTATUS = 0x00000000;
    6 pub const STATUS_UNSUCCESSFUL: NTSTATUS = 0xC0000001u32 as i32;
    7 pub const STATUS_ACCESS_DENIED: NTSTATUS = 0xC0000022u32 as i32;
    8 pub const STATUS_DLL_NOT_FOUND: NTSTATUS = 0xC0000135u32 as i32;
    9 pub const STATUS_ENTRYPOINT_NOT_FOUND: NTSTATUS = 0xC0000139u32 as i32;
   10 pub const STATUS_INVALID_IMAGE_FORMAT: NTSTATUS = 0xC000007Bu32 as i32;
   11
   12 // 3. 你的内部自定义 Enum (零开销, 无字符串)
   13 #[derive(Debug, Clone, Copy, PartialEq, Eq)]
   14 #[repr(u32)]
   15 pub enum HypnusError {
   16     Success = 0,
   17     SyscallFailed,
   18     ModuleNotFound,
   19     SymbolNotFound,
   20     InvalidPeFormat,
   21     MemoryProtectionFailed,
   22 }
   23
   24 // 4. 关键点：实现从 HypnusError 到 NTSTATUS 的转换桥梁
   25 impl From<HypnusError> for NTSTATUS {
   26     fn from(err: HypnusError) -> Self {
   27         match err {
   28             // 将内部错误精准映射为合法的系统级错误码
   29             HypnusError::Success => STATUS_SUCCESS,
   30             HypnusError::ModuleNotFound => STATUS_DLL_NOT_FOUND,
   31             HypnusError::SymbolNotFound => STATUS_ENTRYPOINT_NOT_FOUND,
   32             HypnusError::InvalidPeFormat => STATUS_INVALID_IMAGE_FORMAT,
   33             HypnusError::MemoryProtectionFailed => STATUS_ACCESS_DENIED,
   34             //
      兜底方案：其他所有未专门映射的内部错误，统一对外报告为常规失败
   35             _ => STATUS_UNSUCCESSFUL,
   36         }
   37     }
   38 }

  总结：这套架构为什么是顶级免杀的最终形态？

  通过这三步组合：
   1. 内部 Enum 替代 anyhow：消灭了堆分配，消除了字符串。
   2. stealth_bail! 宏：保留了 Debug 模式下的强力调试输出，同时保证了 Release
      模式下的绝对零特征。
   3. From<Enum> for NTSTATUS
      映射表：在向外暴底结果时，抛弃内部类型，完美伪装成一个规规矩矩的底层
      Windows 组件。如果发生了错误，EDR
      或分析师看到的只是一个类似于“系统常规访问拒绝”的
      0xC0000022，而不是什么“Hypnus Payload Failed to Inject”。

  这套架构既满足了你深入底层的要求（不依赖 alloc，兼容 C
  ABI），又提供了工业级的可维护性和极限的 OpSec。puerto
  的作者正是基于这样的考量，才在 error.rs 中留下了那段 NTSTATUS
  转换代码的草稿（虽然由于项目完成度的原因被注释掉了）。



你的担忧非常专业且直击痛点。你说的“影响正常的系统返回值”，确实是
  dinvk / uwd / puerto 这种底层系统调用包装器/解析器必须考虑的绝对核心问题。

  我们需要严格区分“框架自身的错误”与“底层 OS API
  的业务返回状态”。如果不搞清楚这两者的界限，强行把它们混为一谈映射成
  NTSTATUS，确实会带来灾难性的逻辑混乱。

  让我来为你深度梳理为什么 puerto 原作者注释掉了那个 NTSTATUS
  转换，以及在这种底层项目中，到底该如何完美处理返回值。

  ---

  一、 核心痛点：为什么不能把所有错误都映射为 NTSTATUS？

  你的工具（如 puerto / dinvk）通常充当着调用代理的角色。

  比如，你想调用 NtAllocateVirtualMemory，这个过程分为两个阶段：
   1. 解析阶段（框架层操作）：在 ntdll.dll 中根据哈希寻找这个函数的地址。
   2. 执行阶段（系统层操作）：拿着解析到的地址，传递参数，让内核执行。

  错误重叠与语义混淆的灾难
  假设你在第一步（解析阶段）找不到这个函数的地址，根据我们之前的方案，你可以返回
  一个伪造的 0xC0000139 (STATUS_ENTRYPOINT_NOT_FOUND)。这看起来没问题。

  但问题出在第二步（执行阶段）：
  假如你成功找到了函数的地址，并调用了它。内核执行后失败了，返回了原生的
  0xC0000005 (STATUS_ACCESS_VIOLATION)。

  这时候，如果你用一个统一的 Result<NTSTATUS, HypnusError> 或 Enum Error
  来包装，你就会面临极其尴尬的境地：
   * 外部调用者收到了一个 NTSTATUS。
   * 他根本无法分辨这个 NTSTATUS
     是你（也就是工具层）因为没找到模块伪造发出的，还是目标系统函数在底层执行完
     毕后真实返回的业务状态。
   * 更严重的是，有些系统 API 成功执行时的返回值并不是 0，而是一些警告性的
     NTSTATUS（比如
     STATUS_IMAGE_NOT_AT_BASE）。如果你把工具自身的错误强制转型并丢给调用者，调
     用者如果按照 OS API 的逻辑去处理，就会发生完全不可预期的行为。

  这就是为什么 puerto 作者最终注释掉了 From<Error> for NTSTATUS。
  因为作为一个底层的 API
  Invoker，篡改或模拟业务层的状态码是一种“越权”行为，会破坏调用链的透明度。

  ---

  二、 业界顶级工具是如何解决这个冲突的？

  针对这种“既要有框架自身错误，又不能污染原生 API
  返回值”的需求，顶级红队代码有两种经典架构：

  架构 A：严格分离（使用 Option 或双层返回） —— puerto 正在做的事

  你在前面检索出的 puerto 源码注释揭示了作者的思路：

   1 // 禁用result<>,改用option<*mut c_void>
   2 // 关于是option好还是*mut
     c_void好,需要进一步分析,option在底层还是有极少的语言特性的,*mut
     c_void是最隐蔽的

  对于解析器（Resolver）部分，它不返回 Result/NTSTATUS，只返回指针：

   1 fn get_proc_address(...) -> Option<*mut core::ffi::c_void> {
   2     // 成功：返回 Some(地址)
   3     // 失败：打印 debug_log!，然后直接返回 None 甚至 null_mut()
   4 }

  在宏或包裹层：

   1 // 你提供的宏源码：
   2 if address.unwrap().is_null(){
   3     None // 解析阶段失败，宏返回 None
   4 } else {
   5     // 解析成功，直接执行底层函数。
   6     // 底层函数返回什么（比如 NTSTATUS），这个宏就原封不动地返回什么。
   7     Some(unsafe{func($($arg),*)})
   8 }
  评价： 这个设计非常清晰。工具层（是否找到函数）通过 Option / Null 指针体现；OS
  业务层（系统调用成功与否）通过内部包裹函数的返回值原封不动地透传。二者互不干扰
  。

  架构 B：复合状态返回（高级设计）

  如果你的项目不仅是解析函数，本身也有大量复杂的业务逻辑（就像 hypnus
  要进行多次栈替换、创建事件），你可以采用 “状态指针透传” 的模式。这种模式在 C
  语言的免杀框架中极其常见。

  你的所有核心框架函数返回我们之前设计的自定义 Enum
  Error（零开销，控制流），但如果你的函数内部调用了原生 API，且你想把原生的
  NTSTATUS 传递给调用者以便诊断，你可以通过参数指针把真实状态传出来：

    1 // 定义你的底层错误枚举
    2 #[derive(Clone, Copy, PartialEq, Eq)]
    3 #[repr(u32)]
    4 pub enum HypnusError {
    5     Success = 0,
    6     ApiResolutionFailed,
    7     SyscallFailed, // 注意这个
    8 }
    9
   10 // 核心逻辑函数：
   11 // 返回值 HypnusError 代表框架执行路径是否正常。
   12 // 参数 `out_status` 用于装载发生错误时，底层操作系统真实返回的 NTSTATUS。
   13 fn inject_payload(out_status: &mut NTSTATUS) -> HypnusError {
   14
   15     // 1. 尝试获取 API（框架层错误）
   16     let Some(p_alloc) = get_api(...) else {
   17         stealth_bail!(HypnusError::ApiResolutionFailed, "Api not found");
   18     };
   19
   20     // 2. 调用真实 API（系统层交互）
   21     let status = unsafe { p_alloc(...) };
   22     if status != STATUS_SUCCESS {
   23         // 如果系统 API 报错，把真实的 NTSTATUS 存入传出参数
   24         *out_status = status;
   25
   26         // 框架返回自身状态：SyscallFailed
   27         stealth_bail!(HypnusError::SyscallFailed, "Syscall failed with
      {:#x}", status);
   28     }
   29
   30     HypnusError::Success
   31 }

  为什么架构 B 很强？
   * 界限分明：HypnusError::ApiResolutionFailed
     发生时，调用者知道是你的哈希没对上；HypnusError::SyscallFailed
     发生时，调用者可以立刻去检查 out_status，里面原汁原味地保存着 0xC0000005。
   * 不污染环境：它不再自作聪明地把“找模块失败”映射成系统的“DLL没找到”，保留了
     OS 交互的纯洁性。

  总结

  你的记忆是对的，不能随意使用 From<Error> for NTSTATUS
  来处理那些充当系统调用代理的模块返回值。

  综合考虑，puerto 目前走向了架构 A（利用 Option 甚至裸指针 *mut c_void 结合
  debug_log! 宏静默丢弃错误日志）。这在仅仅充当 dinvk 的场景下是最优解：不引入
  Result 的语言特性，最大程度接近底层 C 汇编。

  而对于业务逻辑更复杂的 hypnus，如果要替代 anyhow::bail!，强烈建议你采用架构
  B（自定义 Error Enum + stealth_bail! 宏 + 传出参数保存真实
  NTSTATUS）。这既解决了 anyhow
  堆分配的问题，又完美保持了对系统真实底层状态的追踪能力。



# 变量的名称

在红队工具中直接使用和源项目相同或者具有高度明显危险语义的单词,是否会暴露,有哪些方式避免?

**正确配置了编译选项的前提下，Rust源码中写的变量名和内部函数名，对免杀性（OpSec）没有任何影响**.Rust 是一门 AOT（Ahead-of-Time）编译语言  
它的编译流水线是：Rust 源码 -> HIR-> MIR -> LLVM IR -> 汇编 -> 机器码当代码被编译为机器码时,局部变量量变成了栈上的偏移量（如 `[RBP-0x20]`）和寄存器（如RCX）；函数调用变成了计算好的内存地址跳转（如 call 0x7FF7B0012345）。EDR 扫描的是最终的机器码，它根本不知道你源码里给变量起了什么中二的名字  
但
1. 未剥离符号表 (Unstripped Symbols):Rust 默认在 Release 构建中也会保留部分符号表,使用逆向工具(IDA等)会出现函数名.
  * 解决方案:在 Cargo.toml 中配置
```rust
[profile.release]
strip = true //或者 strip = "symbols"，
```
2. Panic 字符串泄露 (Panic Formatting):如果你在代码里用了 .unwrap()、.expect("NtCreateThreadEx Failed")或者发生了数组越界。Rust 为了打印友好的崩溃信息，会把发生 Panic的源码文件路径和函数名硬编码到二进制里。EDR 会轻易发现C:\Users\...\dinvk-puerto\src\injector.rs 这样的路径
```rust
[profile.release]
   2         panic = "abort" //发生严重错误直接让进程闪退，绝不打印回溯信息
```

3. 导出函数名 (Exported Functions):如果你正在写一个 DLL 供某些机制加载，你用了 `#[no_mangle]` pub extern "system" fn ReflectiveLoader()，这个名字会作为 DLL的导出表（Export Table）明文保存。
  * 解决：使用毫无意义的名字（如 DllMain,  UpdateChecker），或者使用纯序号导出（Export by Ordinal）

4. 明文字符串调用 API (Plaintext API Names)
  * 后果：虽然你变量名起得好，但如果你写了 GetProcAddress(h, "NtAllocateVirtualMemory\0".as_ptr())，这个字符串依然会作为只读数据（.rdata）存在二进制里，这是高危 IOC



> 做好了正确的编译期配置（strip = true, panic = "abort", lto = true）之后，在 Rust源码中定义的任何变量名、函数名、结构体名，在最终编译出的 Release版二进制文件中都会灰飞烟灭。 它们只存在于编译器的AST（抽象语法树）和中间表示（MIR）阶段，因此你可以随心所欲地使用dinvk_syscall_allocate这样直白、高危的变量名，这完全不影响免杀性，反而极大地提升了你的代码可维护性

> 真正的致命威胁是“硬编码的明文字符串”。 无论是 println!("NtCreateThreadEx Failed")，还是简单的变量赋值 let api_name ="VirtualAlloc"，只要字符串是明文写入的，它就会原封不动地存储在最终 PE 文件的.rdata（只读数据段）或 .data 段中。杀软的静态特征码扫描（YARA rules）扫的就是这些段

> 关于 debug_log! 中的明文字符串：debug_log!("明文")。之所以在 puerto中它是安全的，唯一的原因就是宏内部包裹了`#[cfg(debug_assertions)]`这个条件编译属性告诉Rust 编译器：“如果是 Release 构建，请把大括号里的代码直接从语法树上砍掉，当它完全不存在。”因此，在 Release 编译下，debug_log! 后面的明文字符串连进入 .rdata 段的资格都没有，直接在编译前期被丢弃了。这是极其优雅的安全机制


# 关于s!宏(obfstr)是否足够安全

**obfstr原理**  
编译时对字符串XOR对齐加密,在运行时动态解密

**局限性**  
1. 动态分析有痕迹:以s!("message")为例,静态分析中搜不到明文message.但EDR可识别XOR解密桩的汇编模式.代码中如果有大量这种解密模式,会增加熵值,被标记为高度可疑.  
2. 运行时内存暴露:字符串在运行时解密后,会以明文形式存在栈或堆内存中.如果EDR在解密后\返回错误前的瞬间抓取了内存快照,字符串依然会暴露
3. 日志:即使混淆字符串,如果该错误最终打印到控制台或OutputDebugString输出,EDR会从系统调用监控中捕获明文.

> puerto中解决了输出到控制台的问题,通过macros.rs的debug!,在其中自定义了一个栈缓冲区,避免了alloc,但debug!使用了OutputDebugStrign这个函数.和自定义NTSTATUS的不同用途
> debug!用于调试,在release阶段可以抹去痕迹,自定义NTSTATUS是运行时必须用到的

**Error Enum优势**

已经彻底消除解密桩指纹,完全适用no_std环境

puerto中error的error enum已经使用这种形式

此外:  
实现统一的NTSTATUS映射,自定义的Enum可以通过`impl From<Error> for i32`完美伪装成windows原生NTSTATUS.这样函数返回值在调用栈和寄存器中模拟windows 系统内核返回的错误,达到极致的伪装

**优化方案**

1. 不使用字符串改用使用自定义的错误枚举.在puerto的error.rs中已经使用了.这样:
* 完全无字符串特征,在调试中通过内部的错误代码表(error.rs的enum)中查看调试
* 只依赖core不依赖alloc及std.可运行在用户态/内核态/引导加载程序
* 极低的汇编熵值:汇编时只转为简单的mov eax,0xC0000002; ret 没有任何解密逻辑.

1. 关于“29位自定义错误 (0xE...)”的实战评估

  它的本质： 它是 Windows 协议中预留给开发者的“私有领地”。

   * 优点（逻辑安全性）： 绝对不会与 Windows 原生错误码（如
     0xC...）发生冲突。这对于你的 uwd
     内部逻辑跳转非常重要，你可以清晰地知道：这个错误是我自己抛出的，还是系统底
     层抛出的。
   * 缺点（隐蔽性风险）：
     正如你之前提到的，它是一个特征。虽然它比字符串隐蔽，但如果一个进程频繁返回
     0xE... 这种在常规软件中极少见的错误码，它会成为启发式扫描的一个加分项。

  ---

  2. 更高级的替代方案：仿生伪装 (Mimicry)

  如果追求极致的隐蔽性，我认为比“29位置1”更合适的是：直接使用 Windows
  原生的、含义相近的错误码（29位保持为0）。

   * 方案：
       * 没找到 Gadget？返回 0xC000000F (STATUS_NOT_FOUND)。
       * 解析 PEB 失败？返回 0xC0000005 (STATUS_ACCESS_VIOLATION)。
       * 参数检查失败？返回 0xC000000D (STATUS_INVALID_PARAMETER)。
   * 为什么这更好？
       * 完全消融： 你的错误码在海量的系统日志和 EDR
         监控中，看起来与普通的、写得烂的系统驱动程序或 Buggy
         的应用程序产生的错误一模一样。
       * 对抗分析： 逆向工程师搜不到你的错误码定义，他会认为这些错误是 Windows
         系统抛出的，从而误导他的分析方向。

  ---

  3. 你的日志模式 (OutputDebugStringA) 的位置

  它不是生产级错误处理，它是“脚手架”。

   * 不可替代性： 在开发 uwd
     这种复杂的栈伪造工具时，没有日志你根本无法调试汇编层的对齐问题。你的
     macros.rs 实现非常优雅，尤其是栈上 Buffer 的处理。
   * 生产级的红线： 在 Release 编译版本中，这套系统必须彻底物理剔除。
       * 理由：OutputDebugStringA 是 EDR 监控的“诱饵
         API”。调用它就像在潜行时开启手电筒。

  ---

  4. 负责任的最终总结：生产级错误体系的构建建议

  为了让你的 puerto 重构达到武器级水平，我建议按照以下权重构建体系：

  第一优先级：内部状态机 (The Core Logic)
  在函数内部，使用 Result<T, ErrorEnum>。这个 ErrorEnum 在编译为 Release
  时，映射为 Windows 原生的、常见的内核错误码 (如 0xC0000005)。
   * 原因： 模仿是最深的隐藏。

  第二优先级：静默失败 (The Silent Exit)
  对于极其关键、一旦失败就意味着暴露的环节（如 jmp
  前的检查），不要报错，直接退出线程或进程。
   * 原因： 活下去（持久化）比报错重要。

  第三优先级：调试追踪 (The Developer Trace)
  仅在 Debug 模式下启用你实现的 debug_log! 和 OutputDebugStringA。
   * 原因： 磨刀不误砍柴工，但刀磨好了要把磨刀石藏起来。

  ---

  5. 核心回答：到底是 29 位自定义好，还是原生好？

  负责任的回答是：
   * 如果你正在开发一个需要被其他红队成员集成的通用库，使用 29 位自定义错误
     (0xE...) 更好。因为它专业、规范，不会导致集成时的错误码混淆。
   * 如果你正在开发一个直接用于实战对抗、追求极致免杀的 Payload，使用 Windows
     原生错误码 (0xC...) 进行伪装 更好。因为它具有“仿生”特性，能骗过分析人员和
     EDR。


在红队对抗的终极层面，0xC
  开头的错误码并不是最隐蔽的。

  虽然 0xC... 能在协议层面伪装成系统错误，但它有一个天然的弱点：它太“吵”了。

  以下是针对“终极隐蔽性”的深度多维分析，以及比 0xC 更高级的三个对抗策略：

  ---

  1. 隐蔽性的悖论：错误本身就是指纹

  在高级 EDR（如 CrowdStrike,
  SentinelOne）的监测逻辑中，它们不仅看“你是什么”，更看“你在做什么”。

   * 指纹风险： 如果一个未签名的内存模块（你的 Payload）频繁向系统返回
     0xC0000005 (Access Violation) 或 0xC000000D (Invalid
     Parameter)，这本身就是一个巨大的异常行为特征。
   * 安全研究员视角： 分析人员在查看追踪（Trace）时，会习惯性地过滤掉所有的 0
     (Success)，专门盯着那些非零的 STATUS 看。返回 0xC
     相当于是在告诉分析人员：“快来看，我这里出事了！”

  ---

  2. 比 0xC 更隐蔽的方案 A：成功式失败 (Success-based Failure)

  这是红队武器化中的高级技巧：返回一个“看起来成功”但“逻辑上代表失败”的代码。

   * 做法： 使用 Windows 协议中的 Informational (0x4...) 或 Warning (0x8...)
     级别，甚至直接使用 Success (0x0...) 下的非零值。
   * 例子：
       * 返回 0x00000103 (STATUS_PENDING)。
       * 返回 0x00000001 (虽然不是 0，但 NT_SUCCESS 依然判定为真)。
   * 隐蔽逻辑：
       1. 骗过 EDR： 大多数 EDR 的自动化拦截规则只针对 Severity=Error (11)
          的返回码。
       2. 骗过分析人员： 在海量的 0 记录中，一个偶尔出现的 0x00000001
          极容易被当作由于环境微调导致的正常抖动而忽略。

  ---

  3. 比 0xC 更隐蔽的方案 B：仿生环境码 (Contextual Mimicry)

  你的 uwd 在伪装 kernelbase.dll 或 ntdll.dll 中的函数。

   * 做法： 深入研究你所伪装的那个函数的 “负面路径指纹”。
   * 实战逻辑：
       * 如果你伪装的是 VirtualAlloc，那么查阅文档，看看 VirtualAlloc
         在合法情况下最常返回的错误是什么（比如 ERROR_INVALID_ADDRESS）。
       * 不要发明错误码，要借用错误码。 如果你返回的代码与该 API
         原生可能返回的代码序列完全一致，那么在行为分析（Behavioral
         Analysis）中，你就是完美的。

  ---

  4. 比 0xC 更隐蔽的方案 C：静默坍塌 (Silent Collapse)

  这是我们在 puerto
  重构中最推崇的“武器级”做法：不返回任何错误码，而是直接改变程序的后续行为。

   * 逻辑：
       * 如果 uwd 没找到 Gadget，它不报错，而是返回 Success (0)。
       * 但在内部，它会设置一个全局标志位。
       * 后续的恶意逻辑检测到这个标志位后，会自动退化为执行一段完全合法的代码（
         比如弹出一个无关痛痒的系统设置窗口），然后正常结束进程。
   * 效果： 所有的监控日志里，你的程序从头到尾都是“绿色的”（全 0
     返回），完全没有触发任何报警阈值。

  ---

  5. 负责任的最终结论：生产级代码如何选？

  在重构 puerto 和 uwd 时，我的建议是：

   1. 不要迷信 29 位自定义 (0xE...)： 它适合做私有协议，但特征明显。
   2. 不要滥用 0xC： 它是最后的防线（用于崩溃前的最后挣扎），但容易引起注意。
   3. 首选“成功掩盖下的退出”：
       * 内部逻辑：Result<T, InternalEnum>。
       * 外部表现：0x00000000 (Success) 配合 静默自杀（Silent Exit）。

  总结
  极致的隐蔽不是“伪装成错误的系统代码”，而是“伪装成完全正常的系统代码”。

  在 uwd 的解析逻辑中，如果我们发现 IMAGE_RUNTIME_FUNCTION
  找不到了，最专业的做法不是返回
  0xC0000001，而是立刻停止伪装尝试，并让进程像没发生任何事一样平滑地结束。


  1. CET (Control-flow Enforcement Technology) 在 Win11 的现状

  CET 是栈伪造（Stack Spoofing）的“天敌”。 它利用 CPU 硬件维护一个
  影子栈（Shadow Stack），记录每一个 CALL 压入的真实返回地址。一旦执行 RET
  时，硬件会自动对比当前栈与影子栈，如果不一致，CPU 立即触发 #CP
  异常并关停进程。

   * 开启情况：
       * 硬件要求： 需要 Intel 11 代及以上或 AMD Zen 3 及以上 CPU。
       * 系统策略： 在 Windows 11 中，核心隔离（Core Isolation） 下的
         内存完整性（Memory Integrity/HVCI） 默认开启。
       * 关键进程：
         现代浏览器（Chrome/Edge）、关键系统进程（LSASS）、以及开启了编译选项
         /CETCOMPAT 的受保护应用，强制开启 CET。
       * 普及度： 在高性能企业级终端（你的目标环境）中，CET 的开启率正在迅速接近
         100%。

  结论： 在开启了 CET 的环境下，uwd 这种通过修改返回地址实现的栈伪造必然失败。

  ---

  2. 为何要在 CET 背景下考虑 NTSTATUS 伪装？

  既然 uwd 遇到 CET 会崩，我们为什么要讨论错误码？

   * 探测逻辑： 你的工具在执行 uwd 逻辑前，应该先探测 CET 是否开启。
   * 优雅退化（Graceful Degradation）：
       * 如果 CET 开启 -> 不执行伪造 -> 记录一个内部状态 ->
         隐式退出或执行非敏感逻辑。
       * 此时，返回 0, 1, 2 就显得极其高级。

  ---

  3. 重新审视 0, 1, 2：从“不合适”到“顶级伪装”

  我必须纠正之前的观点。在极致隐蔽（Stealth）的视角下，我之前的“规范性”建议确实
  保守了。

  为什么 0, 1, 2 成了最优选？

   1. 骗过自动化过滤：
      所有的扫描器（EDR/SIEM）在处理海量日志时，第一步就是：filter(status !=
  0)。
       * 如果你返回 0x00000001 或 0x00000002：它们在 i32 补码中是正数，完美符合
         NT_SUCCESS。
       * 它们会躲过 99% 的自动化警报，被埋没在“成功但带有一些小提示”的噪音中。

   2. 仿生“已知的成功码”：
      Windows 系统中其实有很多返回非零值但依然代表成功的代码。
       * 0x00000103 (STATUS_PENDING)
       * 0x00000105 (STATUS_REPARSE)
       * 策略： 你的枚举 UwdError::NullAddr 可以直接对应 1 或 2。

# 背景知识-模拟NTSTATUS

实现统一的NTSTATUS映射是高级红队工具必须使用的技术.

1. windows原生NTSTATUS原型  
在Windows内核(NTOSKRNL.EXE)中,NTSTATUS是一个32位无符号整数,但由于历史原因,在c中以LONG(i32)表示.  
在rust中为了与Windows api兼容,以i32表示NTSTATUS.因为rust的原始指针调用FFI时,很多winapi的返回值被定义位isize/i32.使用i32可以方便的处理负数补码,Windows中错误最高位通常是1,在i32中表现为负数,避免在比较时频繁使用as u32强制转换.  
* 在二进制中,对于有符号数(isize/i32等),其最高位(bit 31)是1,在补码表示法中,最高位1的整数被解释为负数
* NTSTATUS定义中,最高的两位(bit 30/31)表示严重级别Severity:00,最高位0,i32中是正数,表示Success;11,最高位1,i32中是负数,表示Error;10,i32中是负数,表示warning
* 这样在c/rust可以通过>=0或<=0(根据NTSTATUS的正负数)判断成功/错误和警告
* 使用i32模拟NTSTATUS的错误类别,本质上利用了cpu的补码特性,将逻辑上的成功/失败转为数学上的正负

2. 必须使用From Trait.当函数准备返回给外部或给汇编桩时,方便使用.into()转换.有了From Trait可以方便的在enum error和i32之间的相互转换,并可以使用rusult的?操作符隐式调用.into()

3. 利用NTSTATUS的第29位(Customer).可以将29位置1,自定义一套格式合法的错误体系.如0xE0000001,对EDR是一个未知的用户定义错误,但对自己是一个有特殊含义的代码.
* 设计一套NTSTATUS
  * 不能直接使用0,1,2这种简单的数字:在NTSTATUS语境下,0,1,2默认都是成功.其最高的两位都是0.在windows内核看来,00代表STATUS_SUCCESS/STATUS_WAIT
  * 如果用2表示函数地址为空,调用者执行if(status>=0即NT_SUCCESS宏)结果会是true.这意味着这个错误被系统/EDR误认为是一个成功的返回结果,导致后续逻辑在错误前提下继续运行.即如果要表达错误,最高位bit 31必须为1
  * EDR会监控敏感的系统错误(如0xC0000001 STATUS_UNSUCCESSFUL).这会增加风险.而
  * 29位微软选择使用0,如果沿用的话会极致隐藏错误和系统原生错误重合,但也会让自己难以分清是Windows系统报的错误还是自定义的错误
  * 29位自定义为1,规避了冲突

4. 引入Enum error建立了自定义的错误翻译层:内部使用Enum,边界处在extern "C"函数或宏的最外层执行转换.为了模拟NTSTATUS,对齐内核风格:Success必须为0;致命错误必须以0xC0开头;资源缺失模仿STATUS_NOT_FOUND(0xC000000F)

5. 通过From Trait实现的NTSTATUS映射,在rust的高层抽象和Windows底层二进制协议搭建桥梁.代码层是rust,运行时模拟了Windows系统组件.



# 关于anyhow库在no_std环境下的适用

尽管在 Cargo.toml 中设置了 default-features = false，但anyhow 并不是完全的 no_std 库:  
1. anyhow 默认依赖于 std::error::Error 特性。在 no_std 模式下，它需要开启alloc 特性才能运行（它使用 `Box<dyn Error>` 存储错误）
2. 如果目标环境是内核驱动/嫉妒首先的payload.甚至连alloc(动态内存分配)都不允许,anyhow将完全无法工作

> puerto中实现了自定义的alloc,能够匹配anyhow在no_std下的运行
>
> 但仍然不要使用anyhow,而是使用NTSTATUS实现完美的混淆
